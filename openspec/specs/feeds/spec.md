# Feeds Specification

> Generated by openlore v1.0.0 on 2026-06-15 15:11

## Purpose

Maintains a thread-safe registry of named feed producers, enabling registration, lookup, and
enumeration to drive the goroutine-per-feed polling platform.

## Requirements

> `internal/feeds/producer.go`

### Requirement: Register

The system SHALL adds a feed producer to the registry keyed by its name(). overwrites any existing producer registered under the same name.

#### Scenario: RegisterNewProducer
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** A registry with no producer under the given name
- **WHEN** Register is called with a producer
- **THEN** The producer is stored and subsequently retrievable by its Name()

#### Scenario: OverwriteExistingProducer
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** A producer is already registered under the same name
- **WHEN** Register is called with a new producer of the same name
- **THEN** The new producer replaces the old one with no error

#### Scenario: ConcurrentRegistrationIsSafe
- **GIVEN** Multiple goroutines registering producers simultaneously
- **WHEN** Register is called concurrently
- **THEN** All writes complete without data races and no producer is lost

### Requirement: Get

The system SHALL retrieves a registered producer by name. returns the producer and a boolean indicating presence.

> Implementation: `getStreakSummary` in `recipes/gamification/streak-tracker/handler.ts` · confidence: heuristic

#### Scenario: ProducerFound
- **GIVEN** A producer is registered under the given name
- **WHEN** Get is called with that name
- **THEN** Returns the producer and true

#### Scenario: ProducerNotFound
- **GIVEN** No producer is registered under the given name
- **WHEN** Get is called with that name
- **THEN** Returns nil and false

#### Scenario: ConcurrentReadDuringWriteIsSafe
- **GIVEN** A goroutine is concurrently calling Register
- **WHEN** Get is called simultaneously
- **THEN** Returns a consistent result without data races

### Requirement: All

The system SHALL returns a snapshot of all currently registered producers. order is not guaranteed.

> Implementation: `ALL_FIXED` in `tui/hookwise_tui/tabs/status.py` · confidence: heuristic

#### Scenario: EmptyRegistry
- **GIVEN** No producers have been registered
- **WHEN** All is called
- **THEN** Returns an empty slice

#### Scenario: MultipleProducersRegistered
<!-- openlore-test: tags=smoke,regression (auto) -->

- **WHEN** All is called
- **THEN** Returns one entry per registered producer with no duplicates

#### Scenario: SnapshotIsolation
- **GIVEN** All is called while another goroutine is registering a producer
- **WHEN** All returns
- **THEN** The returned slice reflects a consistent point-in-time view without data races

The system SHALL fetches current weather from open-meteo api using configured lat/lon coordinates, decodes the wmo weather code into emoji and description, and returns a feedenvelope. returns a fallback result on any error (network failure, bad status, decode error, missing data).

- **THEN** Returns a FeedEnvelope containing temperature, emoji, and description fields

#### Scenario: HTTPRequestFailure
<!-- openlore-test: tags=regression (auto) -->

- **WHEN** Produce is called

#### Scenario: JSONDecodeFailure

- **THEN** Returns a fallback FeedEnvelope and logs the missing data reason

### Requirement: SetFeedsConfig

The system SHALL receives the full feeds configuration and stores latitude and longitude for use in the open-meteo api request url. implements the configaware interface.

#### Scenario: ConfigWithLat/lon
- **GIVEN** FeedsConfig contains latitude and longitude values
- **WHEN** SetFeedsConfig is called

#### Scenario: ConfigWithoutCoordinates
- **GIVEN** FeedsConfig does not include lat/lon fields

### Requirement: FallbackResult

The system SHALL constructs a safe fallback feedenvelope with '--' placeholder values instead of zero values (e.g. 0°f) so the tui renders an obvious non-value rather than misleading data. logs the reason for the fallback.

#### Scenario: FallbackOnError
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** Any error condition during weather fetch
- **WHEN** fallbackResult is called with a reason string
- **THEN** Returns a map with placeholder '--' values and the reason is logged

### Requirement: WeatherTestFixture

The system SHALL returns a static map representing a valid weather feed payload for use in tests. keys must match the production produce output to prevent silent field-name drift between producer and consumer.

#### Scenario: FixtureKeyParity
- **GIVEN** WeatherTestFixture is called
- **WHEN** The returned map keys are compared to live Produce output
- **THEN** All keys match the production envelope to prevent cross-boundary schema drift

### Sub-component: UrlConstruction

> Implements: `fmt.Sprintf`

Constructs the Open-Meteo API endpoint URL by interpolating latitude and longitude into the request template string.

#### Requirement: InterpolateCoordinates

The system SHALL produce a fully-formed url string by substituting latitude and longitude float values into the open-meteo endpoint template; the resulting url must contain both coordinate values as query parameters.

#### Scenario: ValidCoordinates
- **GIVEN** a URL template containing two %f placeholders and latitude=37.7749, longitude=-122.4194
- **WHEN** fmt.Sprintf is called with the template and those coordinates
- **THEN** the returned string contains '37.774900' and '-122.419400' as query parameter values

#### Scenario: ZeroCoordinates
- **GIVEN** latitude=0.0 and longitude=0.0 (null island)
- **WHEN** fmt.Sprintf is called
- **THEN** the returned URL contains '0.000000' for both parameters without omitting them

### Sub-component: FallbackResultBlock

> Implements: `fallbackResult`

Produces a safe feed envelope containing '--' sentinel values so the TUI never renders misleading zeros when the Open-Meteo API is unavailable.

#### Requirement: LogFallbackReason

The system SHALL log the provided reason string at an appropriate level via core.logger before constructing the fallback envelope, so operators can diagnose why live data was unavailable.

#### Scenario: NetworkError
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** the HTTP client returned a connection-refused error
- **WHEN** fallbackResult is called with reason='connection refused'
- **THEN** a log entry containing 'connection refused' is emitted before the function returns

#### Requirement: ProduceSentinelEnvelope

The system SHALL return a feed envelope whose data map contains '--' for temperature and condition fields; the envelope must be constructed via newenvelope to maintain schema consistency and must not contain numeric zero values that could be mistaken for real readings.

#### Scenario: SentinelTemperature
- **GIVEN** any non-empty reason string
- **WHEN** fallbackResult returns
- **THEN** the data map's temperature field equals '--' and no numeric weather value is present

#### Scenario: EnvelopeStructure
- **GIVEN** fallbackResult is called
- **WHEN** the returned map is inspected
- **THEN** it contains exactly the three top-level keys required by NewEnvelope (type, timestamp, data)

### Sub-component: EnvelopeConstruction

> Implements: `NewEnvelope`

Centralises feed envelope creation to enforce a stable three-key schema (type, timestamp, data) and prevent field name drift between Go producers and the Python TUI consumer.

#### Requirement: EnforceThreeKeySchema

The system SHALL produce a map with exactly three top-level keys—'type', 'timestamp', and 'data'—regardless of what the caller passes as data; no additional keys must appear at the top level.

#### Scenario: ExactlyThreeKeys
- **GIVEN** feedType='weather' and a non-empty data map
- **WHEN** NewEnvelope is called
- **THEN** the returned map has exactly three keys: 'type', 'timestamp', 'data'

#### Scenario: TypeFieldPreserved
- **GIVEN** feedType='weather'
- **WHEN** NewEnvelope returns
- **THEN** envelope['type'] == 'weather'

#### Requirement: StampTimestamp

The system SHALL populate the 'timestamp' field with the current utc time via time.now at the moment of envelope construction; the value must be a non-zero time so the tui can compute data age.

#### Scenario: TimestampNonZero
- **GIVEN** NewEnvelope is called at any point in time
- **WHEN** the returned envelope's 'timestamp' value is inspected
- **THEN** the timestamp is non-zero and falls within one second of the call time

#### Requirement: PassDataUntouched

The system SHALL place the caller-supplied data map under the 'data' key without modifying, filtering, or adding fields to it, so the producer retains full control over data field names.

#### Scenario: DataFieldsPreserved
- **GIVEN** data map contains keys 'temperature', 'emoji', 'description'
- **WHEN** NewEnvelope returns
- **THEN** envelope['data'] is the same map with all three keys intact and no extra keys added

### Sub-component: WgAdd

> Implements: `(*sync.WaitGroup).Add`

Increments the WaitGroup counter before each goroutine is launched so that wg.Wait blocks until all concurrent fetchItem calls complete.

#### Requirement: IncrementBeforeLaunch

The system SHALL call wg.add(1) synchronously in the launching goroutine before the goroutine that calls wg.done is started, preventing a race where wait returns before all goroutines finish.

#### Scenario: AddCalledBeforeGoroutineStart
- **GIVEN** N story IDs are to be fetched concurrently
- **WHEN** the producer iterates the ID slice
- **THEN** wg.Add(1) MUST be called exactly once per goroutine, synchronously before go fetchItem(...) is
invoked

#### Scenario: CounterMatchesDoneCount
- **GIVEN** 5 story IDs are fetched
- **WHEN** all goroutines complete
- **THEN** the total number of wg.Add(1) calls MUST equal the total number of wg.Done() calls

### Sub-component: WgDone

> Implements: `(*sync.WaitGroup).Done`

Decrements the WaitGroup counter when a fetchItem goroutine finishes, ensuring wg.Wait unblocks only after all concurrent fetches are complete.
