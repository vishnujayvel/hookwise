# Notifications Specification

> Generated by openlore v1.0.0 on 2026-06-15 15:11

## Purpose

Evaluates system state across three domains (budget, guard effectiveness, coaching) and creates
deduplicated daily notifications when alert thresholds are crossed.

## Requirements

> `internal/notifications/producers.go`

### Requirement: RunAll

The system SHALL orchestrates all notification producers in sequence, collecting and joining errors without short-circuiting so all checks run regardless of individual failures.

#### Scenario: AllChecksPassWithNoAlerts
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** Cost is below threshold, no guard rules triggered ≥5 times, and coaching alert level is none
- **WHEN** RunAll is called
- **THEN** No notifications are created and nil is returned

#### Scenario: MultipleProducersTrigger
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** Budget is exceeded and coaching alert is active
- **WHEN** RunAll is called
- **THEN** Both budget and coaching notifications are created and nil is returned

#### Scenario: OneProducerErrors,OthersStillRun
<!-- openlore-test: tags=smoke,regression (auto) -->
- **GIVEN** Budget check returns an error but coaching check succeeds
- **WHEN** RunAll is called
- **THEN** Coaching notification is still created and the budget error is returned via errors.Join

#### Scenario: NilStatesAreSafe
- **GIVEN** costState and coachState are nil
- **WHEN** RunAll is called
- **THEN** Budget and coaching producers no-op and nil is returned

### Requirement: CheckBudget

The system SHALL compares today's total spend in coststate against the configured threshold and creates a budget_threshold notification if exceeded, with deduplication ensuring at most one notification per calendar day.

#### Scenario: ThresholdExceeded,NoPriorNotificationToday
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** TotalToday exceeds threshold and no budget_threshold notification exists for today
- **WHEN** CheckBudget is called
- **THEN** A notification is created containing the actual spend and threshold values

#### Scenario: ThresholdExceededButAlreadyNotifiedToday
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** TotalToday exceeds threshold and a budget_threshold notification already exists for today
- **WHEN** CheckBudget is called
- **THEN** No new notification is created and nil is returned

#### Scenario: ThresholdNotExceeded
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** TotalToday is below threshold
- **WHEN** CheckBudget is called
- **THEN** No notification is created and nil is returned

#### Scenario: NilOrZeroThresholdGuard
- **GIVEN** costState is nil or threshold is zero or negative
- **WHEN** CheckBudget is called
- **THEN** Function returns nil immediately without querying or writing

### Requirement: CheckGuardEffectiveness

The system SHALL queries the analytics events table for pretooluse events grouped by tool name, and creates a guard_effectiveness notification for each tool that has triggered at least 5 times today, with per-tool deduplication.

#### Scenario: ToolTriggered5+Times,NotYetNotified
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** A tool has ≥5 PreToolUse events today and no guard_effectiveness notification exists for it
- **WHEN** CheckGuardEffectiveness is called
- **THEN** A notification is created identifying the tool name and block count

#### Scenario: ToolTriggered5+TimesButAlreadyNotified
<!-- openlore-test: tags=smoke,regression (auto) -->
- **GIVEN** A tool has ≥5 PreToolUse events today and a guard_effectiveness notification already exists
containing that tool name
- **WHEN** CheckGuardEffectiveness is called
- **THEN** No duplicate notification is created

#### Scenario: ToolTriggeredFewerThan5Times
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** All tools have fewer than 5 PreToolUse events today
- **WHEN** CheckGuardEffectiveness is called
- **THEN** No notifications are created and nil is returned

#### Scenario: MultipleToolsTriggerSimultaneously
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** Two distinct tools each have ≥5 PreToolUse events today, neither previously notified
- **WHEN** CheckGuardEffectiveness is called
- **THEN** One notification is created per qualifying tool

#### Scenario: NilDbGuard
- **GIVEN** db is nil
- **WHEN** CheckGuardEffectiveness is called
- **THEN** Function returns nil immediately

### Requirement: CheckCoaching

The system SHALL inspects the coachingstate alert level and creates a coaching_prompt notification with level-specific human-readable content, deduplicated per alert level per day.

#### Scenario: YellowAlert,NotYetNotifiedToday
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** CoachingState.AlertLevel is 'yellow' and no coaching_prompt notification exists for that level today
- **WHEN** CheckCoaching is called
- **THEN** A notification is created with yellow-specific wording referencing tooling minutes and mode

#### Scenario: OrangeAlert,NotYetNotifiedToday
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** CoachingState.AlertLevel is 'orange' and no coaching_prompt notification exists for that level today
- **WHEN** CheckCoaching is called
- **THEN** A notification is created with orange-specific wording suggesting a review pause

#### Scenario: RedAlert,NotYetNotifiedToday
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** CoachingState.AlertLevel is 'red' and no coaching_prompt notification exists for that level today
- **WHEN** CheckCoaching is called
- **THEN** A notification is created with red-specific wording strongly recommending stopping

#### Scenario: AlertLevelAlreadyNotifiedToday
<!-- openlore-test: tags=smoke,regression (auto) -->
- **GIVEN** CoachingState.AlertLevel is 'orange' and a coaching_prompt notification for 'orange' already exists
today
- **WHEN** CheckCoaching is called
- **THEN** No duplicate notification is created and nil is returned

#### Scenario: AlertLevelIsNoneOrEmpty
<!-- openlore-test: tags=smoke (auto) -->
- **GIVEN** CoachingState.AlertLevel is 'none' or empty string
- **WHEN** CheckCoaching is called
- **THEN** No notification is created and nil is returned

#### Scenario: NilCoachStateGuard
- **GIVEN** coachState is nil
- **WHEN** CheckCoaching is called
- **THEN** Function returns nil immediately

### Requirement: FormatCoachingContent

The system SHALL produces a human-readable coaching notification message tailored to the alert level (yellow/orange/red), embedding tooling minutes and current mode from coachingstate.

#### Scenario: YellowLevelMessage
- **GIVEN** AlertLevel is 'yellow'
- **WHEN** formatCoachingContent is called
- **THEN** Returned string contains 'yellow', tooling minutes, and a 'step back' suggestion

#### Scenario: OrangeLevelMessage
- **GIVEN** AlertLevel is 'orange'
- **WHEN** formatCoachingContent is called
- **THEN** Returned string contains 'orange' and 'review pause' language

#### Scenario: RedLevelMessage
- **GIVEN** AlertLevel is 'red'
- **WHEN** formatCoachingContent is called
- **THEN** Returned string contains 'red' and 'strongly recommend stopping' language

#### Scenario: UnknownLevelFallback
- **GIVEN** AlertLevel is an unrecognized value
- **WHEN** formatCoachingContent is called
- **THEN** Returned string includes the raw alert level, current mode, and tooling minutes

### Requirement: HasNotificationToday

The system SHALL queries the notifications table to determine whether a notification of a given producer and type was already created on the current utc calendar day.

#### Scenario: NotificationExistsForToday
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** A row exists in notifications matching producer, type, and today's date prefix
- **WHEN** hasNotificationToday is called
- **THEN** Returns true and nil error

#### Scenario: NoNotificationExistsForToday
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** No matching row exists in notifications for today
- **WHEN** hasNotificationToday is called
- **THEN** Returns false and nil error

### Requirement: HasNotificationTodayWithContent

The system SHALL queries the notifications table to determine whether a notification of a given producer, type, and content substring was already created on the current utc calendar day, using like-escaped content matching.

#### Scenario: MatchingNotificationExistsWithContentSubstring
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** A row exists matching producer, type, today's date, and the content substring
- **WHEN** hasNotificationTodayWithContent is called
- **THEN** Returns true and nil error

#### Scenario: NoMatchingNotificationWithThatContent
<!-- openlore-test: tags=regression (auto) -->
- **GIVEN** Rows exist for producer/type/today but none contain the given content substring
- **WHEN** hasNotificationTodayWithContent is called
- **THEN** Returns false and nil error

#### Scenario: ContentSubstringContainsSQLLIKEWildcards
- **GIVEN** contentSubstr contains '%' or '_' characters
- **WHEN** hasNotificationTodayWithContent is called
- **THEN** Wildcards are escaped so the query performs literal substring matching
